# PayPulse — Financial Logic Specification & Audit

**Generated 2026-04-23** for the fresh re-onboarding scenario (user: `torresjosebryan@gmail.com`). Documents the *current* implementation after 5 turns of fixes, the mapping between the brief's proposed model and the actual schema, and the known divergences. Read top-to-bottom before resetting your account.

---

## 1. Current Architecture — One-Paragraph Summary

PayPulse stores a **single `User.currentBalance`** that acts as the rollover seed. Every dashboard request recomputes today's spendable via `balance = currentBalance + thisPeriodIncome − thisPeriodExpenses − thisPeriodBills − thisPeriodPlans`. The engine is in `utils/financeEngine.js`. The API lives in `routes/summaryRoutes.js` (`/paycheck-current` + `/projected-balance`). "Today" is pinned to **America/Los_Angeles** on the server (Vercel runs UTC; PT would otherwise flip pay-period boundaries a day early) and on the client (`Intl.DateTimeFormat` with `timeZone: "America/Los_Angeles"`). The payday cron no longer mutates `currentBalance`; income is added live by the formula.

## 3. Onboarding Flow (current + correct)

1. User enters initial bank balance → written to `User.currentBalance`.
2. User picks pay frequency → `IncomeSource.frequency`.
3. User enters next payday → `IncomeSource.nextPayDate`. **This is the biweekly/weekly anchor and is never recalculated.**
4. User enters recurring paycheck amount → `IncomeSource.amount`; `isPrimary: true`; `isActive: true`.
5. `User.incomeType` is `"fixed"` by default; user can toggle to `"variable"` in Settings.
6. `User.createdAt` is the onboarding timestamp (Mongoose auto-set).

After onboarding, every transaction has a `date` (or `dueDate`/`paidDate`/etc.) stored as a Date with date-only semantics.

## 4. Pay Period Algorithm

**Implementation:** `utils/paycheckUtils.js` → `getCurrentPayPeriod({ lastPaycheckDate, frequency, targetDate })`.

**Biweekly example** (anchor `2026-04-24`):

| Target date | Period returned |
| --- | --- |
| 2026-04-23 | `{ start: 2026-04-10, end: 2026-04-23, nextPayDate: 2026-04-24 }` |
| 2026-04-24 | `{ start: 2026-04-24, end: 2026-05-07, nextPayDate: 2026-05-08 }` |
| 2026-05-07 | `{ start: 2026-04-24, end: 2026-05-07, nextPayDate: 2026-05-08 }` |
| 2026-05-08 | `{ start: 2026-05-08, end: 2026-05-21, nextPayDate: 2026-05-22 }` |
| 2026-04-20 | Walks backward from anchor — returns the period containing 2026-04-20. |

**Semi-monthly weekend rule (intentional):** if a computed payday falls on Saturday, shift to Friday. If on Sunday, shift to preceding Friday. Never forward to Monday. This is NOT in the brief's simple `floor(days / 14)` algorithm but is a product requirement — preserved in `adjustToFriday()` and exercised by `getCurrentPayPeriod`/`getPaydaysInRange`.

**Twicemonthly** → 1st and 15th of each month, same Fri adjustment.

**Monthly** → same day-of-month as anchor, month-by-month.

## 5. Balance Formula (current, authoritative)

File: [routes/summaryRoutes.js](../routes/summaryRoutes.js), endpoint `/api/summary/paycheck-current`.

```
balance = currentBalance
        + periodTotals.totalIncome    // paydays in [periodStart, periodEnd] × amount + one-time income in period
        − periodTotals.totalExpenses  // Expense docs with date in [periodStart, periodEnd],
                                      // excluding savings and autoGenerated
        − totalBillsDue               // sumBillsInPeriod(bills, periodStart, periodEnd, overrideMap, payments)
                                      // includes paid-within-period (via paidAmount) and paid-early cross-period
        − totalPaymentPlansDue        // paid installments by paidDate, unpaid installments by scheduled date
```

**Window is `[periodStart, periodEnd]` for every outflow term.** A prior revision used `[today, periodEnd]` for bills, which silently dropped bills the user had paid earlier in the same period. Corrected this turn.

**What counts once, under this formula:**

- Income for every payday in the period — once.
- User-logged Expense docs in the period — once (savings and autoGenerated excluded).
- Bills with `dueDate in period` and unpaid — once (reserve).
- Bills with `dueDate in period` and paid with `paidDate in period` — once (via `paidAmount`).
- Bills with `dueDate outside period` but `paidDate in period` — once (paid-early cross-period loop in `sumBillsInPeriod`).
- Plan installments with `date in period` and unpaid — once.
- Plan installments with `paidDate in period` (paid-early OR paid-on-time) — once.

**What is NOT counted:**

- Paid bills whose `paidDate` is outside the period (they belong to a different period).
- Auto-generated bookkeeping Expense rows (`autoGenerated: true`) — excluded by `sumExpensesInPeriod`'s filter and by `/api/expenses`'s default query.
- Savings transactions — already mutated `currentBalance` at transaction time; adding them to the formula would double-count.

## 6. Savings (critical — current behavior)

Savings deposits and withdrawals mutate `User.currentBalance` directly:

- Deposit $100 → `currentBalance -= 100`, `SavingsGoal.savedAmount += 100`.
- Withdraw $50 → `currentBalance += 50`, `SavingsGoal.savedAmount -= 50`.

The brief's formula adds `+savingsDeposits − savingsWithdrawals` as separate terms. **Do not do this.** Under the current code those deltas are already inside `currentBalance` — adding them again produces the wrong sign and double-counts the transfer. If a future refactor decides to stop mutating `currentBalance` on savings events, the formula will need the additive terms then; not before.

**Savings are never counted as bills, plans, or expenses.** Expense docs with `category: /^savings$/i` are filtered out of every spending sum via `sumExpensesInPeriod`'s regex filter.

## 7. Bills (current behavior)

Schema: [models/Bill.js](../models/Bill.js) + [models/BillPayment.js](../models/BillPayment.js) + [models/PaymentOverride.js](../models/PaymentOverride.js).

A Bill is a **recurring template** (`dueDayOfMonth` + `amount` + optional `startDate`/`lastPaymentDate`). Each instance of the bill is materialized by computing "day-of-month in month" per period. A `BillPayment` row records when a specific instance was paid (composite unique index on `user, bill, dueDate`).

**Marking paid** (via [utils/billPaymentService.js](../utils/billPaymentService.js)):

- Upserts a `BillPayment` with `paidDate` and `paidAmount`.
- Sets `Bill.paid: true` (mirror field for per-bill UI state).
- **Does not** create an auto-Expense anymore (removed this turn). Paid-early cross-period outflows are now picked up by `sumBillsInPeriod`'s paid-early loop, so no mirror Expense is needed. Legacy auto-Expense rows in the DB are excluded from aggregates by `autoGenerated: { $ne: true }` and from `/api/expenses` responses (opt in with `?includeAuto=true` for debugging).

**Unmarking paid** deletes the `BillPayment` AND any legacy auto-Expense matching the description pattern (for backward compat with old data).

**Filtering rules, concretely:**

- Unpaid bill, due in period: counted via effective-amount path.
- Paid bill, due in period, paidDate in same period: counted via `paidAmount`.
- Paid bill, due in period, paidDate BEFORE period: not counted in this period (it belongs to the earlier period).
- Paid bill, due in period, paidDate AFTER period: not counted in this period (it belongs to the later period — paid late).
- Paid bill, due OUTSIDE period, paidDate INSIDE period (paid-early cross-period): counted via `paidAmount` in the paid-early loop.

## 8. Payment Plans (current behavior)

Schema: [models/PaymentPlan.js](../models/PaymentPlan.js). Each `plan.payments[]` element: `{ id, date, amount, paid, datePaid, paidDate, paidEarly }`.

**Counting rule (updated this turn):** bucket by when the money moved.

- `paid: true` → bucket by `paidDate` (or `datePaid`). Counted in whatever period `paidDate` falls in, whether paid-early or paid-on-time. (Previously the code excluded paid-on-time under the assumption `currentBalance` had already absorbed the outflow — but plan pays never mutate `currentBalance`, so those installments were silently dropped. Fixed.)
- `paid: false` → bucket by scheduled `date` (future outflow reserved for).

**`paidEarly` flag** is set at mark-paid time: `entry.paidEarly = todayYMD < scheduledYMD`. Stored but no longer drives different math — just a bookkeeping field for UI labels.

## 9. Expenses (current behavior)

Simple: each Expense has `date`, `amount`, `category`, `description`, optional `autoGenerated`. No `paid` concept.

- Savings deposits logged as `category: "Savings"` Expense rows are filtered out of every spending aggregate by regex (`$not: /^savings$/i`).
- Auto-generated rows (`autoGenerated: true`) are excluded from `sumExpensesInPeriod` and from `/api/expenses`'s default response. No new code creates these; legacy rows are invisible to users.

**Duplicate prevention:** not enforced at the DB level. `scripts/duplicate-audit.js` (this turn) detects same-day/same-amount/same-category/same-description duplicates and reports them.

## 10. Income (current behavior)

Two sources:

- `IncomeSource` — recurring. `frequency` + `amount` + `nextPayDate` anchor. `getPaydaysInRange(anchor, frequency, rangeStart, rangeEnd)` generates the paydays in any window.
- `OneTimeIncome` — single dated amounts (bonuses, reimbursements, savings withdrawals also write here).

**For the current period formula:** `periodTotals.totalIncome = recurring(paydays-in-period × amount) + oneTime(dated-in-period)`.

**The payday cron** (`api/cron/payday.js` + `jobs/paydayIncome.js`) runs at 09:00 UTC daily and stamps `IncomeSource.lastAutoIncomeDate`. It **no longer increments** `currentBalance`. If a future refactor re-enables the `$inc`, the formula must drop the income term to avoid double-counting — choose one.

## 11. Timezone (America/Los_Angeles, pinned)

- Server: `routes/summaryRoutes.js` exports helpers `todayInAppTz()` and `resolveToday(req)`. Every endpoint that uses "today" calls `resolveToday(req)`, which honors `?localDate=YYYY-MM-DD` then falls back to `todayInAppTz()` (LA-pinned via `Intl.DateTimeFormat("en-CA", { timeZone: "America/Los_Angeles", ... })`).
- Client: `client/src/context/DataCache.jsx`'s `todayParam()`, `client/src/hooks/useCurrentPaycheckSummary.js`, `client/src/pages/Calendar.jsx`, and `client/src/components/OverdueCheckModal.jsx` all use `Intl.DateTimeFormat` with `America/Los_Angeles` to build the local date sent as `?localDate=`. Browser-local `new Date()` is not used for period math.
- If the user is outside LA, they still see LA "today." That's a product decision (single source of truth for the app's pay-period calendar), not a default.

Memory rule `feedback_server_tz_vercel.md` locks this in across future sessions.

## 12. The brief's Part 11 walkthrough — reconciled

The brief's walkthrough double-subtracts a bill due on "today" and mishandles the savings sign. This doc's formula avoids both:

- Bills/plans bucket by the period their money moved in, and the dashboard formula counts them once.
- Savings are already in `currentBalance`; the formula does not add them.

**Expected behavior, under the current code, for a fresh onboarding scenario like the brief's Part 11:**

```
Onboarding 2026-04-23 — currentBalance = $500, anchor = 2026-04-24, frequency = biweekly
Period 1 = [2026-04-24, 2026-05-07]

Data in period 1 (unpaid):
  Apple Care  — due 2026-04-24 — $19.99
  Claude      — due 2026-04-27 — $100.00
  Nike plan   — due 2026-04-24 — $50.00
  Groceries   — expense 2026-04-25 — $30.00
  Gas         — expense 2026-04-26 — $45.00
  Paycheck    — 2026-04-24 — +$1,131.17 (income)
  Savings     — deposit 2026-04-25 → currentBalance decremented to $400

Dashboard balance on 2026-04-27 (PT):
  = 400 (currentBalance, already reflects the $100 savings deposit)
  + 1131.17 (period income)
  − 75.00 (period expenses: groceries + gas)
  − 119.99 (period bills: Apple Care + Claude, both unpaid)
  − 50.00 (period plans: Nike installment 1, unpaid)
  = 1,286.18
```

`currentBalance` is $400 not $500 because the $100 savings deposit already came out of it. If you want the brief's exact arithmetic ($1,485.18 etc.), note it does not subtract savings from spendable — that's the sign bug documented in §6.

## 13. Verification steps (manual — run after re-onboarding)

Because no MongoDB access is available from this session, the nine verification steps in the brief need to be walked by the user. After re-onboarding:

1. **Timezone.** Load paypulse.money at 10pm PT. Dashboard says "After all bills through [day before next payday]." "Next paycheck" pill shows the correct date. The "Did you pay this?" prompt does not fire a day early.
2. **Period algorithm.** Open Calendar. Current period highlighted matches the anchor + frequency rule. Clicking the next payday's date shows Period 2's snapshot.
3. **Bills filtering.** On a period's day sheet, "Expenses" section lists only user-logged expenses (no bill-shaped entries). Bills total matches the sum of bills in the period (paid+unpaid). Paid-early bills appear in the paid period, not the due period.
4. **Savings logic.** Deposit $100 → dashboard "You can spend" drops by $100 exactly; savings card goes up by $100. Withdraw $50 → spendable up $50, savings down $50.
5. **Early payment.** Mark a bill due next period as paid today. Dashboard balance drops by the bill's amount. Calendar shows the bill as paid on today's date. Next period's snapshot does not include the bill again.
6. **Duplicate audit.** `MONGO_URI=... TARGET_EMAIL=... node scripts/duplicate-audit.js` — report lands in `docs/duplicate-audit.md`. Review; the script never deletes.
7. **Next paycheck pill.** Matches the next payday computed from anchor + frequency. After today passes that date, it flips forward.
8. **Fresh onboarding math.** The numbers on day 1 match the Part 11 example rules (with the savings-sign and double-count corrections from §12).
9. **No regressions.** Savings isolation, case-insensitive category filter, calendar teal for savings, donut palette, expense filter tabs — all still work per the Phase 1 test plan.

## 14. Known limitations (explicitly deferred)

- **Period N > 1 rollforward.** The current code assumes `currentBalance` is either (a) the onboarding seed for period 1 users, or (b) manually kept current. There is no automatic period-to-period rollforward mechanism. For users who are in their second or later period without manual maintenance, the formula may drift. This is a separate architectural improvement (store rollover per-period, or walk the period chain on every request) and is out of scope for today's re-onboard. If you stay on the app past 2026-05-07, we revisit.
- **Duplicate prevention at write time.** The duplicate-audit script catches duplicates after the fact. There's no DB-level unique index on (user, date, amount, description) for expenses. Adding one is easy but not in today's scope.
- **Timezone for multi-user.** All users see America/Los_Angeles. If a second user in NY onboards, their "today" will still be LA. This is the stated product choice; revisit if it becomes a product issue.
